Skill File
Definition: A skill file describes a capability or task the system can perform.

Contents: It usually defines:

The category (e.g., quiz, shopping, comparisons).

The goals (e.g., “generate flashcards,” “find coupons”).

Sometimes specific skill names if they’re modular.

Purpose: Skills are reusable building blocks that can be loaded dynamically depending on the user’s request.

Use Case:

Example: A “quiz” skill file lets the agent generate multiple-choice questions.

Example: A “shopping-savings” skill file enables finding coupons or cashback offers.

Think of a skill file as a toolbox entry — it tells the agent what tools exist and how to use them.

Agent Definition File
Definition: An agent definition file describes the agent itself — its identity, scope, and orchestration logic.

Contents: It typically includes:

Agent name/ID

Capabilities (skills it can load)

Policies and constraints (e.g., safety rules, refusal boundaries)

Interaction style (tone, response format, memory usage)

Purpose: It’s like the blueprint of the agent — defining how it should behave, what skills it can call, and how it interacts with users.

Use Case:

Example: A “Travel Agent” definition file might specify it can load skills for flight booking, hotel search, and reminders.

Example: A “Study Agent” definition file might specify quiz + flashcards + practice-test skills.

How to Create Agents
Creating an agent usually involves three steps:

Define the Agent File

Give it a name, scope, and purpose.

Specify which skills it can access.

Set policies (e.g., refusal rules, tone, formatting).

Attach Skills

Link skill files that provide capabilities.

Example: Attach “quiz” + “flashcards” to a “Study Agent.”

Deploy & Test

Run the agent in a sandbox or environment.

Test with sample queries to ensure it loads the right skills and responds correctly.

building agents = writing the agent definition file + creating skill files + testing/deploying via a runtime

What Happens When You “Load” an Agent
It’s not like an OS process in RAM. Instead:

Agent definition file → parsed into a structured object (Python dict, JSON, YAML).

Framework (Semantic Kernel / AutoGen) → loads this object into memory (your laptop’s RAM).

LLM → is invoked with a system prompt that includes:

Agent description (persona).

Skills/tools (capabilities).

Policies/refusal rules.

👉 The LLM doesn’t become the agent; it behaves like one because the orchestration framework wraps its inputs/outputs with the agent definition.

Analogy with OS Processes
OS process in RAM → binary code loaded, executed by CPU.

Agent orchestration → structured text (YAML/Markdown) parsed, injected into LLM prompt, executed by the LLM model.

Execution location:

Parsing + orchestration → happens in your laptop’s CPU/RAM (Python runtime).

Language generation → happens inside the LLM (local model or cloud API).

So instead of “loading into RAM like an OS process,” it’s more like:

Framework loads metadata into RAM.

Framework sends metadata + user query as a prompt to the LLM.

LLM generates output conditioned by that metadata.

Who Loads It, Where It Runs
Loader → The orchestration framework (Semantic Kernel, AutoGen).

Where loaded → In your laptop’s runtime memory (Python objects).

Execution → The LLM (local or cloud) processes the prompt and generates text.

Processor involvement → Your CPU/GPU runs the orchestration code; the LLM inference may run locally (if you use Ollama/LM Studio) or remotely (Azure OpenAI, Anthropic, etc.).

Best for runtime execution and testing flows.

YAML Execution Flow
File Structure: YAML is a strict data serialization format.
Parsing

Frameworks (Semantic Kernel, AutoGen, LangChain) use YAML parsers (PyYAML, ruamel.yaml) to directly convert this into a Python dict / object.

No extra parsing logic needed — YAML → native object.

Loading

The parsed object is stored in runtime memory (RAM) as structured metadata.

Framework knows immediately: this is an agent definition because of the root key agent:.

Execution

Framework injects fields (name, description, skills) into the system prompt for the LLM.

Skills are mapped to callable functions or prompt templates.

LLM generates output conditioned by this metadata.

👉 YAML = direct execution format. Minimal parsing overhead, deterministic structure.

Markdown Front‑Matter Execution Flow
File Structure

Markdown front‑matter is metadata embedded in a doc.

Parsing

Frameworks must first detect the front‑matter block (--- ... ---).

Then parse it as YAML‑like key/value pairs.

This requires an extra parsing step (using libraries like python-frontmatter).

Loading

Parsed metadata is stored in RAM as a dict.

Framework interprets fields differently:

tools: instead of skills: → maps to capabilities.

model: → tells which LLM to use.

color: → UI hint, not execution logic.

Execution

Framework builds a system prompt using description.

Tools are mapped to available functions.

LLM is instructed to behave as the agent.

The rest of the Markdown file (outside front‑matter) can hold human‑readable PRD content.

How Does the LLM Get Picked in Deployment?
In deployment, you don’t hard‑code the LLM in a test script. Instead, you use configuration:

Environment variables → e.g., LLM_MODEL=gpt-4, LLM_API_KEY=xxxx.

Config files → JSON/YAML that specify the default LLM service.

Orchestration framework settings → Semantic Kernel or AutoGen reads these configs at startup.

👉 So the deployed environment picks the LLM based on runtime configuration, not the test script.

Execution Flow in Deployment
Agent files loaded → Framework parses YAML/Markdown into memory.

Skills loaded → Framework registers capabilities.

LLM binding → Framework looks at environment/config (e.g., GPT‑4 via Azure OpenAI).

Runtime execution → User queries routed through the agent → framework constructs prompt → sends to chosen LLM.

Output returned → Response delivered via API, web app, or service.

“In local test scripts, I explicitly bind agents to LLMs in code. But in deployment, we don’t ship those test scripts. Instead, we deploy the agent/skill definition files and the orchestration framework. The LLM is picked at runtime based on configuration — environment variables or service settings. So the framework loads the agent metadata, then binds it to whichever LLM is specified in the deployment config. That separation makes the system flexible and future‑proof.”

Test scripts = explicit binding for dev. Deployment = config‑driven binding at runtime.

How to Narrate the Diagram
1. Start with the Application Layer
“At the top we have the application layer — APIs like /catalog and /checkout that handle user queries from the frontend.”

2. Move into the Orchestration Runtime (Center)
“Those queries flow into the orchestration runtime, which is the brain of the system. It has four key functions: agent parser & loader, skill registry, prompt builder, and LLM router with governance.”

3. Explain Inputs (Left Side)
“On the left, the runtime loads agent definitions — for example, CatalogAgent and CheckoutAgent — and skill modules like inventory tools and payment skills. These are parsed from YAML or Markdown files.”

4. Explain LLM Binding (Right Side)
“On the right, the runtime binds to whichever LLM backend is specified in configuration — GPT‑4 via Azure OpenAI, Claude via Anthropic, or local Hugging Face/Ollama models. This binding is dynamic, based on environment variables or service settings.”

5. Explain Outputs (Bottom)
“Finally, the runtime returns structured responses back to the application layer and frontend — product recommendations, price comparisons, checkout confirmations — all governed and logged for safety.”
